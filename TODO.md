## Текущее состояние репозитория (снимок)

Уже есть **сквозной сценарий**: регистрация → тест (Likert / binary / forced_choice) → summary → лента людей (`/feed`) → профили/сравнение → лайки → диалоги/чаты → **Threads-like коммуникации**: лента общения `/timeline` (посты/ветки), темы как вкладка/фильтр, ответы с gating по ценностям (`/posts/{id}/can-reply`), лайки/репост/цитата.

- **Стек зафиксирован:** FastAPI + SQLAlchemy + Alembic, Next.js 14 + Tailwind, JWT, SQLite dev / PostgreSQL prod.
- **Сущности в БД:** `User`, `Question`, `QuestionAxis`, связи вопрос–ось, `Answer`, лайки, чаты/сообщения, **`thread_posts`** (корень+ответы), **`thread_post_topics`** (темы/оси), **`thread_post_likes`**. Исторические `discussion_*` перенесены в `thread_posts` миграцией (см. Alembic `0016`).
- **Матчинг v0:** `backend/app/core/matching.py` — оси из ответов через `question_axis_link`; сравнение только по осям с данными у обоих; инсайты по порогам расстояния.
- **Сиды:** `backend/seed.py` — оси и онбординг; `backend/scripts/seed_fixture_users.py` — симуляция N пользователей (по умолчанию 100) со **случайными** ответами (`--count`); пароль демо см. `README.md` / константа `CHAOS_PASSWORD` в скрипте.
- **Фронт:** единая **дизайн-система** (`mm-*` в `frontend/app/globals.css`), Geist, нижняя навигация с активным разделом; **поллинг таймлайна** с ETag и **ленивая** загрузка списка осей только на вкладке «Темы»; на чате/ленте людей/группе — **параллельная** загрузка независимых запросов где возможно (`Promise.all`); **`lib/api.ts`** — автоповторы GET при 502/503/504 и сетевых сбоях (удобно при cold start на Render).
- **Производительность API:** лента, групповой матчинг и блок «общее в комнате» используют **`compute_user_axis_scores_batch`**; списки постов тредов (таймлайн, детали, ответы, **`GET /users/{id}/threads`**) — **`_build_posts_out`** в `thread_posts.py` (единый батч без N+1).
- **Пробелы относительно идеала MVP:** OAuth, деплой и метрики, второй пак вопросов, PWA — см. фазы ниже.

## Фаза D — качество и запуск

- [ ] **Доступность:** не полагаться только на цвет (частично: подписи ✓/≠ в ленте); аудит контраста — по желанию; **фокус/лейблы/sr-only** после дизайн-обновления — summary + таймлайн (итерация).
- [x] **Сиды для демо:** `seed_fixture_users.py` (массовая симуляция со случайными ответами).
- [ ] **Деплой:** staging + прод; HTTPS; бэкапы БД.
- [ ] **Метрики:** завершили тест, лента, лайк, сообщение.
- [x] **Фронт:** `next/image` (аватары ленты и др., `unoptimized` где нужен произвольный origin); **ETag + If-None-Match** для опроса таймлайна (304 при отсутствии изменений); **ретраи GET** при шлюзовых ошибках (`lib/api.ts`).
- [x] **Коммуникации “как Threads” (ядро):** единый `thread_posts`, лента `/timeline`, ветки, быстрые реакции (like/repost/quote), темы как вкладка/фильтр, value-gating на ответы.
- [x] **Перф тредов:** батч-сборка `ThreadPostOut` (`_build_posts_out`), профиль **`/users/{id}/threads`** на том же коде, что таймлайн.
- [ ] **Коммуникации (улучшение):** уведомления (SSE/polling), контекст «в ответ на…» в активности, удаление/скрытие тредов, поиск по тредам, подписки/уведомления.
- [ ] **Быстрый и “чистый” хостинг:** подобрать площадку/архитектуру деплоя с короткими доменами и заметно меньшей латентностью (регион ближе к пользователям, быстрый cold-start, нормальный Postgres pool/коннекты). Зафиксировать выбор и обновить `README.md` (deploy) под новый стек.
